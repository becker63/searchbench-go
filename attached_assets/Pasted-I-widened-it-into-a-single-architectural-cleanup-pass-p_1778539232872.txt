# Architectural cleanup pass after making `internal/app/round` the single entry point

## Goal

Clean up the remaining architectural issues after making `internal/app/round.Run` the canonical app-layer entry point.

The previous cleanup correctly removed `internal/app/evaluation` and public `internal/app/compare` as sibling app workflows. `compare` now lives privately under `internal/app/round/internal/compare`, and CLI round execution now goes through `internal/app/round`.

This issue should now make the internals match the architecture.

The target architecture is:

```text
surface/cli
  -> app/round
      -> pure models
      -> ports
      -> adapters
```

The central lifecycle should be mechanically real:

```text
ResolveGame
ResolveRound
EvaluateMatches
BuildEvidence
EvaluateObjective
Decide
ProposeNextChallenger
WriteBundle
```

Each phase should actually do what its name says.

## Background

`round.Run` now has the right outer shape, but the implementation still carries some modeling-phase scaffolding:

1. `EvaluateMatches` still behaves like a wrapper around an older “evaluation run” blob.
2. Some phase functions mostly unwrap data that was already produced earlier.
3. `app/round` still owns fake evaluator/scorer/graph behavior that should be behind explicit fake adapters or test fixtures.
4. `app/round` imports surface rendering code for `round-report.txt`.
5. Dataset loading is still effectively fake and embedded in round resolution instead of behind a match-source seam.
6. `internal/app/optimizer` remains a sibling app workflow even though it should be a round phase implementation.
7. Optimizer bundle writing and Python validation are adapter-shaped but currently live in app code.
8. Some pure vocabulary packages are still aliases/placeholders around `domain`.
9. Pkl artifact schema names still bake round numbers into field names.
10. `internal/app/logging` is support/surface infrastructure, not round orchestration.

This pass should clean up these architectural seams before real MCP, dataset, LangSmith, provider, and visualization integrations land.

## Architecture rules

### Single app story

There is only one executable app story:

```text
Run a round.
Everything else is a phase of that round.
```

Do not reintroduce `internal/app/evaluation`.

Do not reintroduce public `internal/app/compare`.

Do not introduce another sibling public app workflow.

### Layering

Preserve these boundaries:

```text
internal/pure
  deterministic SearchBench model only

internal/app/round
  orchestration and lifecycle composition

internal/adapters
  Pkl, filesystem, subprocess, Eino, fake backends, dataset loading, concrete runtime work

internal/surface
  CLI and human presentation only

internal/testing
  test-only fake worlds
```

Surface code must not define artifact formats, scoring semantics, decisions, or execution contracts.

App code should not own console rendering, subprocess validation, or concrete filesystem bundle mechanics unless there is already a narrow adapter wrapper.

Adapters may be messy. Their job is to translate external systems into SearchBench-owned types.

## Scope

This is a cleanup/refactor issue. It should not add real MCP, LangSmith, real dataset loading, or real provider execution.

The goal is to create clean seams so those integrations have obvious places to land later.

# Part 1: Make `round.Run` phase boundaries mechanically real

## Problem

`round.Run` reads correctly, but some phases still rely on earlier all-in-one evaluation behavior.

Bad shape:

```go
matches, err := EvaluateMatches(...)
evidence, err := BuildEvidence(..., matches)
objective, err := EvaluateObjective(..., evidence, matches)
decision := Decide(..., objective, matches)
bundle := WriteBundle(..., matches)
```

but internally:

```text
EvaluateMatches
  -> runs comparison
  -> builds evidence
  -> evaluates objective
  -> renders report
  -> writes bundle
```

or preserves something close to that older shape.

## Required change

Make each phase actually own its phase.

### `EvaluateMatches`

Should only run incumbent/challenger match executions and return match/report execution data.

It must not:

- build final round evidence
- evaluate the Pkl objective
- write the completed round bundle
- create `COMPLETE`

Target conceptual shape:

```go
func EvaluateMatches(ctx context.Context, resolved Resolved, input Input) (MatchRecords, error) {
    // run comparison / policy executions only
    // return RoundReport or enough execution data to build one
}
```

### `BuildEvidence`

Should actually project match/report data into `score.RoundEvidenceDocument`.

Target conceptual shape:

```go
func BuildEvidence(
    resolvedGame game.Contract,
    resolved Resolved,
    matches MatchRecords,
) (score.RoundEvidenceDocument, error) {
    evidence, err := report.BuildRoundEvidence(matches.Report)
    if err != nil {
        return score.RoundEvidenceDocument{}, err
    }

    // preserve existing game id / round id behavior
    // validate if a Validate method exists

    return evidence, nil
}
```

### `EvaluateObjective`

Should actually evaluate the configured Pkl objective against the evidence.

Target conceptual shape:

```go
func EvaluateObjective(
    ctx context.Context,
    resolved Resolved,
    evidence score.RoundEvidenceDocument,
    matches MatchRecords,
) (*score.ObjectiveResult, error) {
    // materialize temporary current evidence if the scorer still needs an evidence.pkl file
    // call internal/adapters/scoring/pkl
    // validate result
    // do not write final completed bundle
}
```

If temporary `evidence.pkl` materialization is required for the current Pkl scorer, keep it local to this phase. It must not create the final round bundle or `COMPLETE`.

### `Decide`

Should explicitly derive or capture the decision after evidence/objective exist.

For now it is acceptable to preserve the existing simple local decision behavior. The important thing is to avoid reaching through an old fully completed evaluation result.

Target conceptual shape:

```go
func Decide(
    resolvedGame game.Contract,
    resolved Resolved,
    evidence score.RoundEvidenceDocument,
    objective *score.ObjectiveResult,
    matches MatchRecords,
) report.Decision {
    return matches.Report.Decision
}
```

A richer decision rule can remain a follow-up.

### `WriteBundle`

Should be the only phase that writes the completed round bundle.

It should write:

```text
resolved-round.json
round-report.json
round-report.txt, unless disabled
evidence.pkl
objective.json
decision.json
metadata.json
COMPLETE
```

It must be the only phase that creates `COMPLETE`.

Target conceptual shape:

```go
func WriteBundle(
    ctx context.Context,
    resolved Resolved,
    evidence score.RoundEvidenceDocument,
    objective *score.ObjectiveResult,
    decision report.Decision,
    next *optimizer.Record,
    matches MatchRecords,
    input Input,
) (pureround.Record, error) {
    // call bundle fs adapter
}
```

## Delete or defang old all-in-one helpers

Delete any function equivalent to `runEvaluationResolved` if possible.

If some helper logic must remain, split it into narrow helpers:

```go
runComparison(...)
renderRoundReportText(...)
writeRoundBundle(...)
currentObjectiveEvidenceRef(...)
materializeCurrentEvidenceForObjective(...)
```

Do not leave any helper that owns the whole lifecycle.

## Tests for this part

Add or update tests:

- `TestEvaluateMatchesDoesNotWriteComplete`
- `TestBuildEvidenceBuildsFromMatchReport`
- `TestEvaluateObjectiveDoesNotWriteComplete`
- `TestWriteBundleWritesComplete`
- existing `round.Run` integration test still writes expected artifacts

Important invariant:

```text
COMPLETE exists only after WriteBundle.
```

# Part 2: Remove `surface/console` dependency from `app/round`

## Problem

`internal/app/round` should not import `internal/surface/console`.

Surface owns CLI/human presentation. App owns orchestration. A rendered report artifact is fine, but the renderer should not force app to import surface.

## Required change

Remove any direct import from:

```text
internal/app/round -> internal/surface/console
```

Choose one of these solutions.

### Preferred solution

Move report text rendering into an adapter:

```text
internal/adapters/report/console
```

or:

```text
internal/adapters/report/text
```

Then `app/round` depends on a narrow renderer interface or adapter.

### Acceptable solution

Define a small interface in `app/round`:

```go
type ReportRenderer interface {
    Render(report.RoundReport) (string, error)
}
```

Default wiring can provide the existing console renderer.

## Non-goals

Do not rewrite the visual style.

Do not redesign terminal output.

Only fix the dependency direction.

## Acceptance criteria

- `internal/app/round` does not import `internal/surface/console`.
- Human report text artifact behavior is preserved.
- CLI output behavior is preserved.

# Part 3: Move fake evaluator/scorer/graph/provider behavior out of `app/round`

## Problem

`internal/app/round` still contains fake local behavior such as fake evaluator model, fake repo tool, fake graph provider, fake scorer, or fake decider.

This was good for proving the model, but it is now dangerous because future integrations may extend fake code instead of replacing it behind proper seams.

## Required change

Move fake runtime behavior behind explicit fake adapters or test fixtures.

Use this rule:

### If fake mode is a supported local/dev backend from Pkl manifests

Put it under:

```text
internal/adapters/backend/fake
```

or more specific packages:

```text
internal/adapters/dataset/fake
internal/adapters/scoring/fake
internal/adapters/codegraph/fake
```

### If fake mode is only for tests

Put it under:

```text
internal/testing/fakes
```

Production code must not import `internal/testing`.

## Preserve current local fake round

The manifest-driven local fake round must still run.

The existing fake backend can remain supported, but it should be clearly adapter-shaped, not app-core-shaped.

## Acceptance criteria

- `internal/app/round` no longer defines fake model/scorer/graph/provider implementations directly.
- Fake behavior lives under `internal/adapters/.../fake` or `internal/testing/...`.
- Existing local fake manifest still works.
- No production code imports `internal/testing`.

# Part 4: Introduce a match-source seam

## Problem

Dataset loading is still embedded in round resolution through fake match construction.

Real LCA integration should not be added yet, but there should be a seam for it.

## Required change

Introduce a narrow app-owned or port-owned interface for loading matches from the resolved dataset spec.

Conceptual shape:

```go
type MatchSource interface {
    LoadMatches(ctx context.Context, spec DatasetSpec) (domain.NonEmpty[domain.MatchSpec], error)
}
```

or, if using existing types:

```go
type MatchSource interface {
    LoadMatches(ctx context.Context, resolved Resolved) (domain.NonEmpty[domain.MatchSpec], error)
}
```

Use existing project types where possible.

The current fake match loader should become one implementation of this seam.

Suggested placement:

```text
internal/ports/matches
```

or, if keeping small for now:

```text
internal/app/round/match_source.go
```

with concrete fake implementation in:

```text
internal/adapters/dataset/fake
```

## Rules

- External dataset vocabulary may use “task,” but SearchBench-facing values should be `domain.MatchSpec`.
- Do not implement the real LCA adapter in this issue.
- Do not add Hugging Face or network access in this issue.

## Acceptance criteria

- Round resolution/evaluation no longer hardcodes fake matches directly.
- A fake match source exists behind an explicit seam.
- Future LCA adapter has an obvious place to plug in.
- Existing local fake round still works.

`internal/app/optimizer` remains a sibling app workflow. But architecturally, optimizer behavior is a phase of `round.Run`: it proposes a `NextChallenger`.

internal/app/round/internal/optimizer

### Acceptable

## Acceptance criteria

- Optimizer is not presented as a second public app workflow.
- `round.Run` remains the only executable app story.
- Evaluation-only rounds still do not require optimizer configuration.
- Optimizer runs only when explicitly configured.

# Part 6: Move optimizer filesystem and subprocess work out of app

## Problem

Optimizer bundle writing, staging directories, checksums, Python `py_compile`, and `os/exec` validation are adapter-shaped work.

App should coordinate next-challenger proposal generation. It should not own concrete filesystem lifecycle or subprocess validation mechanics.

## Required change

Move concrete optimizer filesystem/subprocess behavior into adapters.

Suggested shape:

```text
internal/adapters/optimizer/fs
  bundle writer / staging / metadata / COMPLETE

internal/adapters/policy/python
  py_compile validation
```

or another clear adapter placement.

`internal/app/round/internal/optimizer` or `internal/app/optimizer` should call interfaces/adapters, not directly own subprocess/filesystem details.

## Rules

- Preserve current optimizer behavior.
- Do not redesign optimizer prompt behavior.
- Do not add real provider integration.
- Do not remove Python validation unless tests are updated and rationale is documented.

## Acceptance criteria

- App optimizer code does not directly own `os/exec` Python compilation behavior.
- App optimizer code does not directly own low-level bundle filesystem mechanics.
- Adapter package owns concrete filesystem/subprocess behavior.
- Existing optimizer tests still pass or are updated to target the new adapter boundary.

# Part 7: Decide on pure vocabulary alias packages

## Problem

There are pure packages like:

```text
internal/pure/domain
internal/pure/match
internal/pure/policy
internal/pure/round
internal/pure/game
```

Some are real. Some are thin aliases/placeholders around `domain`.

This creates conceptual drift.

## Required change

Make a small explicit decision and apply only the safe part now.

Choose one:

### Option A: Keep `domain` as the main pure vocabulary package for now

Then delete or reduce placeholder alias packages that do not carry meaningful model behavior.

### Option B: Begin migration toward product vocabulary packages

Move obvious types toward:

```text
internal/pure/match
internal/pure/policy
internal/pure/round
internal/pure/game
```

and shrink `domain` toward shared primitives.

## Recommendation

Prefer minimal churn in this issue.

If a pure alias package is unused and empty except for docs, delete it or leave a clear TODO in package docs.

Do not perform a huge pure model migration unless it is trivial and tests stay small.

## Acceptance criteria

- No meaningless pure alias package remains without a clear purpose.
- If packages are kept, package docs explain why.
- No broad pure model redesign occurs.
- Import-boundary tests still pass.
